# 3. 이벤트 수집 및 모니터링 메커니즘

## 3.1 CSP별 기본 모니터링 도구
각 클라우드 사업자는 통합 로깅 플랫폼을 제공합니다.
- **AWS CloudWatch**: 로그 수집, 메트릭 모니터링, 대시보드 및 알람 제공.
- **Azure Monitor**: 로그 수집(Log Analytics Workspace) 및 통합 인사이트 제공.
- **GCP Cloud Logging**: 분산 시스템의 로그 수집 및 실시간 분석(Log Explorer).

## 3.2 로그 통합 및 중앙 집중화 전략
대규모 환경(Multi-Account/Multi-Region)에서는 로그를 한곳으로 모으는 것이 필수입니다.
- **Log Aggregation**: 여러 계정의 로그를 하나의 전용 보안 계정(Logging Account)으로 집계.
- **Data Lake 구축**: S3나 Google Cloud Storage에 장기 보관하여 비용 절감 및 향후 분석 대비.
- **SIEM 연동**: Splunk, Microsoft Sentinel 등 외부 SIEM(Security Information and Event Management) 플랫폼으로 전송.

## 3.3 실시간 스트리밍 vs 배치 처리
- **실시간 스트리밍 (Real-time)**: 
    - **도구**: AWS Kinesis, Azure Event Hubs, GCP Pub/Sub.
    - **용도**: 심각한 보안 사고(예: 루트 계정 로그인) 발생 시 즉각적인 대응이 필요할 때.
- **배치 처리 (Batch)**:
    - **도구**: AWS Glue, Azure Data Factory.
    - **용도**: 주기적인 감사 보고서 생성, 장기적인 트렌드 분석.

## 3.4 가시성(Observability) 확보의 장애물
- **로그 유실**: 설정 오류로 인한 특정 구간 로그 누락.
- **비용 문제**: 무분별한 모든 로그 수집으로 인한 비용 폭증.
- **로그 파편화**: 구조화되지 않은(Unstructured) 로그로 인한 분석의 어려움 (JSON 포맷 권장).
