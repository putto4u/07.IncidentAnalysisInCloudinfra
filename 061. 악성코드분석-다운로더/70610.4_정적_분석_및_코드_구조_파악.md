# 061. 악성코드분석-다운로더 - 04. 정적 분석 및 코드 구조 파악

## 개요
다운로더 악성코드를 분석하는 데 있어 정적 분석(Static Analysis)은 악성코드를 실행하지 않고 파일 자체의 특성과 코드를 검토하여 정보를 얻는 핵심적인 방법입니다. 이 챕터에서는 PE 헤더 정보 및 문자열 기반의 악성 징후 분석, 그리고 API 해싱 및 동적 로딩 기법을 역공학하는 방법에 대해 학습합니다. 이를 통해 악성코드의 기능을 예측하고 동적 분석 방향을 설정하는 데 필요한 기초 지식을 습득합니다.

## 1. PE 헤더 정보 및 문자열 기반의 악성 징후 분석
Windows 환경에서 실행되는 대부분의 악성코드는 PE(Portable Executable) 파일 형식입니다. PE 파일의 헤더 정보와 내부 문자열을 분석하는 것은 악성 징후를 식별하는 데 매우 효과적인 초기 정적 분석 기법입니다.

### 1.1 PE 헤더 정보 분석
-   **PE 파일이란?** Windows 운영체제에서 실행되는 실행 파일, DLL, SYS 파일 등에 사용되는 파일 형식입니다. 실행에 필요한 다양한 정보(섹션, 임포트/익스포트 함수, 리소스 등)를 담고 있습니다.
-   **분석 도구:** PE-bear, CFF Explorer, Detect It Easy (DIE) 등.
-   **주요 확인 항목:**
    -   **컴파일러/링커 정보:** 어떤 개발 환경에서 컴파일되었는지 확인합니다. (예: Visual C++, MinGW, Go, Delphi 등) 특정 컴파일러는 악성코드와 연관성이 높을 수 있습니다.
    -   **타임스탬프 (Timestamp):** 컴파일된 시간을 확인합니다. 미래 시간으로 설정되어 있거나, 비정상적으로 오래된 시간은 분석가 혼란을 위한 위조일 수 있습니다.
    -   **섹션 정보:** `.text`, `.data`, `.rsrc` 등 표준 섹션 외에 `.UPX0`, `.RLC` 등 패커와 관련된 섹션 이름이 있는지 확인합니다. 실행 가능한 (`.text` 외의) 섹션이 비정상적으로 많은 경우도 의심할 수 있습니다.
    -   **임포트 테이블 (Import Table):** 악성코드가 어떤 시스템 함수(API)를 사용하는지 보여줍니다. 특히 `URLDownloadToFile`, `WinHttpOpen`, `CreateProcess`, `WriteFile`, `VirtualAlloc`, `RegSetValueEx` 등 네트워크 통신, 프로세스 생성, 파일 쓰기, 메모리 할당, 레지스트리 조작과 관련된 함수는 악성 행위와 직접적인 연관이 있습니다.
    -   **버전 정보 / 리소스 섹션:** 파일 설명, 제품명 등 버전 정보가 없거나, 리소스 섹션에 실행 파일(EXE), DLL 등 의심스러운 리소스가 포함되어 있는지 확인합니다.

### 1.2 문자열 기반의 악성 징후 분석
-   **분석 도구:** Strings.exe (Sysinternals), IDA Pro, Ghidra 등.
-   **주요 확인 항목:**
    -   **URL/IP 주소:** C2 서버 주소, 추가 페이로드 다운로드 URL 등이 평문으로 노출되어 있는지 확인합니다. (예: `http://malicious.com/payload.exe`, `192.168.1.100`)
    -   **파일명/경로:** 악성코드가 생성하거나 접근할 것으로 예상되는 파일명 및 경로 문자열을 확인합니다. (예: `C:\Windows\system32\malware.exe`, `%TEMP%	emp.dll`)
    -   **레지스트리 키:** 지속성 확보를 위해 조작할 레지스트리 키 이름 (예: `Software\Microsoft\Windows\CurrentVersion\Run`, `SYSTEM\CurrentControlSet\Services`)
    -   **에러 메시지/디버깅 정보:** 디버깅 관련 문자열(예: `DEBUG`, `TRACE`)이나 에러 메시지는 분석에 도움이 될 수 있습니다.
    -   **패커 관련 문자열:** UPX!, FSG! 등 특정 패커의 시그니처 문자열이 있는지 확인합니다.
-   **주의사항:** 문자열이 난독화되거나 암호화되어 있을 수 있으므로, 평문으로 보이지 않는다고 해서 악성코드가 아니라고 단정해서는 안 됩니다.

## 2. API 해싱 및 동적 로딩 기법 역공학
최신 악성코드들은 탐지 회피를 위해 임포트 테이블에 직접 API 함수 이름을 노출하지 않고, API 해싱이나 동적 로딩 기법을 사용하여 분석을 어렵게 만듭니다.

### 2.1 API 해싱 (API Hashing)
-   **정의:** 악성코드가 사용하려는 API 함수의 이름을 직접 임포트 테이블에 명시하는 대신, API 함수의 이름에 대한 해시 값을 저장하고, 런타임에 이 해시 값과 일치하는 API 함수를 찾아서 동적으로 로드하여 사용하는 기법입니다.
-   **역공학 방법:**
    -   **알고리즘 식별:** 악성코드가 어떤 해싱 알고리즘(예: ROR13, CRC32)을 사용하는지 분석해야 합니다. 이는 주로 코드의 초기 부분에서 해시 계산 루틴을 찾는 방식으로 이루어집니다.
    -   **해시 값 역계산:** 식별된 해싱 알고리즘을 사용하여 알려진 모든 Windows API 함수의 해시 값을 계산하고, 악성코드 내의 해시 값과 비교하여 실제 사용되는 API 함수를 식별합니다.
    -   **디버거 활용:** 디버거를 통해 런타임 시점에서 해시 계산 함수 호출 직전/직후의 값을 확인하거나, API가 실제로 로드되어 호출되는 지점을 추적하여 파악할 수 있습니다.

### 2.2 동적 로딩 (Dynamic Loading) 기법
-   **정의:** `LoadLibrary`와 `GetProcAddress`와 같은 함수를 사용하여 필요한 DLL과 API 함수를 런타임에 동적으로 로드하는 기법입니다. 이는 임포트 테이블에 API 함수가 나타나지 않으므로 정적 분석을 어렵게 만듭니다.
-   **역공학 방법:**
    -   **`LoadLibrary` 및 `GetProcAddress` 호출 추적:** 디스어셈블러(IDA Pro, Ghidra)에서 이 두 함수가 호출되는 지점을 찾고, 어떤 DLL(`LoadLibrary`의 인자)과 어떤 API 함수(`GetProcAddress`의 인자)를 로드하려 하는지 분석합니다.
    -   **문자열 추출:** `LoadLibrary`나 `GetProcAddress`의 인자로 사용되는 DLL 이름(예: `kernel32.dll`)이나 API 함수 이름(예: `CreateProcessA`) 문자열이 난독화되어 있을 수 있으므로, 이를 해독하는 과정이 필요합니다.
    -   **디버거 활용:** 디버거를 사용하여 `LoadLibrary`와 `GetProcAddress` 함수 호출 시 전달되는 인자 값들을 실시간으로 확인하여 실제 로드되는 DLL과 API 함수를 파악합니다.

### 2.3 쉘코드(Shellcode) 분석
일부 다운로더는 매우 작은 크기의 쉘코드를 포함하여 초기 침투 및 추가 페이로드 다운로드를 수행합니다. 쉘코드는 독립적으로 실행되는 작은 바이너리 코드로, 주로 익스플로잇 공격의 페이로드로 사용됩니다.
-   **분석 방법:** 쉘코드는 일반적인 PE 파일 형식이 아니므로, 디스어셈블러에서 Raw Binary로 로드하여 분석하거나, 디버거를 통해 메모리 상에서 실행 흐름을 추적하여 분석합니다.

## 핵심 요약
다운로더 악성코드의 정적 분석은 PE 헤더 정보 및 문자열 분석을 통해 기본적인 악성 징후를 식별하는 것에서 시작합니다. 더 나아가 API 해싱 및 동적 로딩 기법을 역공학하여 악성코드가 실제로 어떤 시스템 함수를 사용하는지 파악해야 합니다. 이러한 분석 결과는 동적 분석 단계에서 중요한 가이드라인을 제공합니다.
