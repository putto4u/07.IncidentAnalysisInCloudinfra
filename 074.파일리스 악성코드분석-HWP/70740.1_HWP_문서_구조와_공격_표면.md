# 1. HWP 문서 구조와 공격 표면 (Attack Surface)

## 1.1 개요
한글(HWP) 문서는 국내 공공기관 및 기업에서 널리 사용되는 워드프로세서 형식으로, 공격자들에게는 매우 매력적인 공격 벡터입니다. HWP 파일은 기본적으로 마이크로소프트의 OLE(Object Linking and Embedding) 구조를 따르는 **Compound File Binary Format (CFBF)**을 기반으로 합니다.

## 1.2 HWP 파일 구조 (CFBF)
HWP 파일은 내부적으로 계층적인 Storage와 Stream 구조를 가집니다. 이는 마치 파일 시스템 내의 폴더와 파일 관계와 유사합니다.

### 1.2.1 주요 Stream 분석
- **FileHeader**: 파일의 버전, 속성, 암호화 여부 등을 정의하는 256바이트 헤더.
- **DocInfo**: 글꼴, 스타일, 문단 모양 등 문서의 전반적인 정보를 담고 있는 스트림.
- **BodyText**: 실제 문서의 본문 내용이 섹션(Section) 단위로 저장되는 스트림.
- **BinData**: 문서 내에 삽입된 이미지, OLE 객체, 포스트스크립트(EPS) 등 바이너리 데이터가 저장되는 핵심 공격 지점.
- **SummaryInformation**: 문서 요약 정보 (작성자, 날짜 등).

## 1.3 주요 공격 표면 (Attack Surface)

### 1.3.1 BinData 스트림의 악용
BinData는 외부 객체를 포함하는 저장소로, 공격자는 이곳에 악성 EPS 파일이나 OLE 객체를 삽입합니다. HWP 엔진이 이 데이터를 렌더링하거나 실행하는 과정에서 취약점이 발생합니다.

### 1.3.2 프리뷰 이미지 (PrvImage/PrvText)
문서 미리보기를 위해 생성되는 데이터로, 파일 탐색기에서 문서를 선택할 때 파싱되는 과정에서 취약점이 악용될 수 있습니다.

### 1.3.3 섹션 데이터 (SectionN)
문서의 각 페이지 정보를 담고 있는 섹션 데이터 파싱 중 버퍼 오버플로우(Buffer Overflow) 취약점이 발생할 수 있습니다. 특히 가변 길이 데이터를 처리하는 루틴이 주요 타겟입니다.

## 1.4 요약
HWP 분석의 시작은 **BinData**와 **DocInfo** 스트림을 면밀히 조사하는 것입니다. 특히 최근의 파일리스 공격은 별도의 실행 파일을 생성하지 않고 메모리 상에서 동작하므로, HWP 엔진의 파싱 로직을 이해하는 것이 필수적입니다.
