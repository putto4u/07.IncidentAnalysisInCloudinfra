# 3. 포스트스크립트(EPS) 악용 기법 심층 분석

## 3.1 EPS (Encapsulated PostScript) 개요
EPS는 어도비에서 개발한 페이지 설명 언어인 PostScript를 기반으로 한 이미지 파일 형식입니다. HWP는 고품질 그래픽 출력을 위해 내부적으로 **Ghostscript** 엔진 등을 사용하여 EPS를 렌더링합니다.

## 3.2 EPS 기반 파일리스 공격 기법
공격자는 EPS 파일 내부에 악성 포스트스크립트 명령어를 삽입합니다. 이는 실행 파일(`*.exe`) 형태가 아니므로 시그니처 기반 백신 탐지를 우회하기 쉽습니다.

### 3.2.1 Ghostscript 취약점 (CVE-2017-8291 등)
EPS 데이터를 처리하는 과정에서 발생하는 메모리 손상 취약점입니다. `/` 명령어를 사용하여 메모리 보호를 해제하고 임의 코드를 실행합니다.

### 3.2.2 난독화 및 복호화 루틴
대부분의 악성 EPS는 쉘코드를 인코딩된 형태로 포함합니다.
- **XOR 난독화**: 특정 키 값으로 쉘코드를 암호화하여 저장.
- **EPS 스크립트 내 복호화**: 포스트스크립트 언어의 루프와 연산 기능을 사용하여 실행 시점에 메모리에서 직접 쉘코드를 복호화.

## 3.3 분석 절차

1.  **스트림 추출**: `HWPScan2` 또는 `oledump.py`를 사용하여 `BinData` 내의 EPS 파일을 추출합니다.
2.  **헤더 확인**: 추출된 파일이 `%!PS-Adobe`로 시작하는지 확인합니다.
3.  **코드 분석**:
    - `/hex` 형태의 대량 데이터 존재 여부.
    - `xor`, `putinterval`, `getinterval` 등 메모리 조작 관련 명령어 탐색.
4.  **복호화 시뮬레이션**: Python 스크립트 등을 작성하여 EPS 내의 인코딩된 데이터를 수동으로 복호화하여 쉘코드 원형을 확보합니다.

## 3.4 최근 동향
최근 한글과컴퓨터 보안 패치로 인해 기본적으로 EPS 실행이 차단되고 있으나, 구버전 한글을 사용하는 환경이나 레지스트리 설정을 통해 강제로 활성화된 환경을 타겟으로 여전히 발생하고 있습니다.

## 3.5 기술적 예시 (Pseudocode)
```postscript
% EPS 내 복호화 루틴 예시
/enc_data <4D5A...원본 데이터> def
/key 0x77 def
0 1 enc_data length 1 sub {
    /i exch def
    enc_data i get key xor
    enc_data i 3 -1 roll put
} for
enc_data exec
```
위와 같은 루틴이 발견된다면 이는 명백한 악성 행위의 징후입니다.
