# 2. 취약점 기반 HWP 공격 분석

## 2.1 HWP 취약점의 역사적 흐름
과거 HWP 공격은 주로 스택 기반 버퍼 오버플로우(Stack-based Buffer Overflow)를 이용했으나, 최신 OS 보안 정책(DEP, ASLR)에 대응하기 위해 **힙 스프레이(Heap Spraying)** 및 **ROP(Return-Oriented Programming)** 기법을 결합한 지능형 공격으로 진화했습니다.

## 2.2 핵심 취약점 유형

### 2.2.1 가변 길이 레코드 파싱 오류
HWP는 데이터를 레코드 단위로 관리합니다. 특정 레코드의 길이를 나타내는 값을 비정상적으로 크게 설정하여 인접한 메모리 영역을 덮어쓰는 방식입니다.
- **대상**: 가변 길이 데이터 처리 함수 (예: `HwpApp.exe` 내 특정 모듈)

### 2.2.2 힙 스프레이 (Heap Spraying)
ASLR(Address Space Layout Randomization)을 우회하기 위해 쉘코드(Shellcode)가 포함된 대량의 데이터를 메모리의 특정 영역에 뿌려놓는 기법입니다.
- **HWP 내 적용**: JavaScript 또는 문서 내 객체 로딩 과정을 통해 쉘코드를 메모리에 배치.

## 2.3 쉘코드 실행 흐름 분석

1.  **트리거**: 사용자가 악성 HWP 파일을 엽니다.
2.  **취약점 실행**: 특정 스트림(예: Section 또는 BinData)을 파싱하는 과정에서 취약점이 발생하여 실행 흐름(EIP/RIP)이 탈취됩니다.
3.  **ROP 체인 실행**: DEP(Data Execution Prevention)를 우회하기 위해 기존 프로그램의 코드 조각(Gadget)을 연결하여 메모리 보호 속성을 변경(예: `VirtualProtect`)합니다.
4.  **쉘코드 동작**: 메모리 보호가 해제된 영역의 쉘코드가 실행됩니다.

## 2.4 분석 포인트
- **EIP 탈취 지점**: 디버거(x64dbg 등)를 통해 예외가 발생하는 시점의 레지스터 상태 확인.
- **메모리 덤프**: 힙 영역을 분석하여 쉘코드의 패턴(`0x90` NOP Sled 등) 탐색.
- **API 후킹**: `CreateProcess`, `URLDownloadToFile`, `WinExec` 등 악성 행위에 사용되는 API 호출 추적.

## 2.5 실습 사례 (추상화)
실제 분석 시에는 `HwpApp.exe`가 특정 `BinData`를 읽어들일 때 `memcpy` 등의 함수에서 크기 검증이 누락되는지 확인하는 과정이 포함됩니다.
